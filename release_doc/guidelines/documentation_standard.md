# 文档记录标准

## 核心要求

**release_doc/ 是 release 工作的唯一权威文档。** 任何接手的同事只看这些文档就能完整执行 release，不需要再问任何人。

## 记录原则

### 1. 消灭不确定性

每个 open item 确认后，不光打勾，还要把以下内容都写清楚：
- **决策内容** — 最终选了什么方案
- **决策理由** — 为什么这样决定
- **具体执行方案** — 怎么做，做到什么程度
- **边界条件** — 哪些情况需要特殊处理，哪些不用管

### 2. 细节完备

包括但不限于：
- 具体的替换示例（如 `/data/rbg/...` → `deepseek-ai/...`）
- 哪些目录/文件不用管，以及**为什么**不用管
- Agent 调查出的建议（如环境变量写法、HuggingFace hub 名）
- 每个方案的优缺点分析

### 3. 可追溯

- 记录决策过程，不只是结果
- 让后续同事理解"为什么这样决定"
- 如果一个决策推翻了之前的理解，要记录为什么改变

### 4. 及时记录

- 用户说的任何决策、偏好、要求，**立即**记录到对应文档
- 不要等到对话结束才批量记录
- 频繁 commit，确保记录不会丢失

### 5. 不遗漏

- Agent 调查的所有发现都要记录
- 用户说的每一条要求都要反映在文档中
- 如果用户说"这个也要记下来"，那**这句话本身也要记录**（因为它定义了记录标准）

## 文档格式规范

### 决策状态标记

每个文档开头标注状态：
- `## 决策状态：已确认` — 所有内容已和用户确认
- `## 决策状态：部分确认` — 部分内容已确认，部分待定
- `## 决策状态：待确认` — 内容是草案，需要用户确认

### 交叉引用

- 使用相对路径链接其他文档
- 不重复内容，只引用
- 示例：`详见 [06_path_cleanup.md](../code_cleanup/06_path_cleanup.md)`
