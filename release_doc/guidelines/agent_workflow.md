# Agent 执行任务的工作流程规范

## 阶段执行模式

每个 release 阶段由一个 agent 独立执行。Agent 执行任务时遵循以下流程：

### 1. 读取阶段文件

- 先读 `stages/` 下对应的阶段文件，了解前置条件、具体步骤、验证标准
- 再读相关的 scope/code_cleanup/components 文档，了解决策细节

### 2. 检查前置条件

- 确认前一阶段已完成
- 确认所需的输入物已就绪

### 3. 执行步骤

- 按阶段文件中的步骤逐一执行
- 遇到不确定的细节时，**不要猜测**，走确认流程（见 [confirmation_protocol.md](confirmation_protocol.md)）
- 每完成一个子步骤，在阶段文件中标记完成

### 4. 验证

- 按验证标准检查输出物
- 运行相关扫描或测试

### 5. 记录

- 更新阶段文件的状态
- 如果发现了新问题或需要新决策，记录到 `tracking/14_open_items.md`
- commit 修改

## 断点恢复机制

**每个 agent 必须遵守**：

1. **开始工作时**：先读 `release_doc/CURRENT_STATUS.md`，了解当前进展到哪了、下一步是什么
2. **工作过程中**：及时更新 `CURRENT_STATUS.md` 中的"已完成"、"未完成"、"下一步行动"
3. **每 2-3 轮对话必须更新一次** `CURRENT_STATUS.md` 并 commit — 这是硬性要求，不是建议。因为 agent 随时可能意外断掉，如果不频繁保存状态，断点恢复就会丢失信息
4. **完成一个 open item 或重要决策后**：立即更新 `CURRENT_STATUS.md` 并 commit
5. **对话结束前**（无论正常结束还是可能中断）：确保 `CURRENT_STATUS.md` 反映最新状态并已 commit

这样即使 agent 意外断掉，下一个 agent 读这个文件就能恢复到断点。

## 关键原则

1. **不跨阶段操作** — 每个 agent 只执行自己阶段的任务
2. **不做未确认的决策** — 所有决策必须在 `tracking/14_open_items.md` 中有记录
3. **频繁 commit** — 每完成一个有意义的子步骤就 commit
4. **细节记录** — 执行过程中发现的任何问题、边界情况都要记录到文档中
5. **确定性高的任务直接执行** — 不需要每件小事都找用户确认，如果方向明确就直接做
6. **非 P0 问题不要停** — 遇到小问题自主决策继续推进，不要因为非关键问题阻塞整体进度
7. **代码隔离原则** — 不修改现有算法核心逻辑，新功能通过配置参数控制，默认关闭，不影响原始代码行为
8. **能让 agent 执行就让 agent 执行** — 主线程尽量做决策和协调，具体执行工作交给子 agent
9. **执行前确认理解** — 执行任何任务之前，先用一两句话跟用户确认自己的理解是否正确

## 代码开发的 Agent 调度策略

### 全程使用 Opus

所有代码开发任务统一使用 **Opus agent**，不使用 Sonnet。

**原因**：Release 代码质量要求极高，一旦出错后续返工成本很大。用 Opus 全程开发虽然更慢更贵，但能从源头保证质量，避免低质量代码进入后续阶段导致连锁问题。

> **历史记录**：早期方案是 Sonnet 写代码 + Opus 审查，后改为全程 Opus。变更原因见上。
