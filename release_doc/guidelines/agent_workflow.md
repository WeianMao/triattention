# Agent 执行任务的工作流程规范

## 阶段执行模式

每个 release 阶段由一个 agent 独立执行。Agent 执行任务时遵循以下流程：

### 1. 读取阶段文件

- 先读 `stages/` 下对应的阶段文件，了解前置条件、具体步骤、验证标准
- 再读相关的 scope/code_cleanup/components 文档，了解决策细节

### 2. 检查前置条件

- 确认前一阶段已完成
- 确认所需的输入物已就绪

### 3. 执行步骤

- 按阶段文件中的步骤逐一执行
- 遇到不确定的细节时，**不要猜测**，走确认流程（见 [confirmation_protocol.md](confirmation_protocol.md)）
- 每完成一个子步骤，在阶段文件中标记完成

### 4. 验证

- 按验证标准检查输出物
- 运行相关扫描或测试

### 5. 记录

- 更新阶段文件的状态
- 如果发现了新问题或需要新决策，记录到 `tracking/14_open_items.md`
- commit 修改

## 关键原则

1. **不跨阶段操作** — 每个 agent 只执行自己阶段的任务
2. **不做未确认的决策** — 所有决策必须在 `tracking/14_open_items.md` 中有记录
3. **频繁 commit** — 每完成一个有意义的子步骤就 commit
4. **细节记录** — 执行过程中发现的任何问题、边界情况都要记录到文档中
5. **确定性高的任务直接执行** — 不需要每件小事都找用户确认，如果方向明确就直接做
6. **非 P0 问题不要停** — 遇到小问题自主决策继续推进，不要因为非关键问题阻塞整体进度
7. **代码隔离原则** — 不修改现有算法核心逻辑，新功能通过配置参数控制，默认关闭，不影响原始代码行为
8. **能让 agent 执行就让 agent 执行** — 主线程尽量做决策和协调，具体执行工作交给子 agent
9. **执行前确认理解** — 执行任何任务之前，先用一两句话跟用户确认自己的理解是否正确

## 代码开发的 Agent 调度策略

### 写代码阶段：Sonnet 为主

每个代码开发子任务：
1. 启动 **Sonnet agent** 执行 2-3 轮代码编写（更快、省钱）
2. 2-3 轮结束后，启动 **Opus agent** 做一轮代码审查：
   - 检查是否有明显错误
   - 检查是否符合开发目标
   - 检查是否符合 release 规范

### 审查后的修复策略

根据 Opus 审查发现的问题严重程度决定修复方式：

| 问题类型 | 修复策略 |
|---------|---------|
| 简单/容易修复的错误 | Sonnet 执行修复 → 再次审查 → 如还有问题继续循环 |
| 偏离较大/复杂的错误 | 方案 A：先让 Sonnet 修完简单的，再让 Opus 修难的 |
| | 方案 B：直接让 Opus 修复 |

### 选择 Sonnet 的理由

- 速度更快
- 成本更低
- 大部分代码编写任务 Sonnet 能胜任
- Opus 的价值在于审查和处理复杂问题，不需要每一行代码都用 Opus 写
