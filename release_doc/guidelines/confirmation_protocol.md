# 与用户确认决策的标准流程

## 核心原则

**一次只讨论一个问题。** 不要一口气抛出多个大问题让用户回答。

## 确认流程

### 第 1 步：调查

在向用户提问之前，先启动 agent 对代码库进行调查：
- 扫描相关代码，了解现状
- 分析影响范围
- 收集具体数据（文件列表、行号、引用关系等）

### 第 2 步：给出方案

基于调查结果，给用户提供：
- **现状总结** — 简洁说明当前是什么情况
- **可选方案** — 列出 2-4 个方案，每个方案写清楚优缺点
- **推荐** — 给出自己的推荐和理由

### 第 3 步：引导确认

- 让用户选择方案或提出修改
- 如果用户的选择涉及需要进一步细化的细节，主动追问
- 主动引导用户确认那些**他们可能没想到但很重要**的边界条件

### 第 4 步：记录

确认后立即：
1. 更新 `tracking/14_open_items.md` — 标记为已确认，记录决策内容和理由
2. 更新相关的详情文档 — 把具体方案写入对应的 md 文件
3. commit 修改

## 反面示例（不要这样做）

- :x: 一次问 7 个问题让用户回答
- :x: 没有调查就直接问用户"你想怎么办"
- :x: 只给一个模糊的建议，没有具体方案
- :x: 确认后忘记记录到文档中
- :x: 记录了决策但没有记录理由

## 正面示例（应该这样做）

- :white_check_mark: "我先启动 agent 调查一下 rkv 包的结构和引用情况，给你几个命名方案选"
- :white_check_mark: 调查完后给出方案表格，每个方案有具体名字、优点、缺点
- :white_check_mark: 用户选了方案后追问："这个方案里 baseline 的包名你倾向叫 kv_compress 还是 kv_compression？"
- :white_check_mark: 确认后更新 14_open_items.md 和 05_repo_structure.md，然后 commit
