## 一、整体架构

```
用户上传文档
    │
    ▼
doc-parser（解析 + 结构化）
    │
    ▼
chunker（智能分块）
    │
    ├─── Embedding (bge-large-zh-v1.5) ──→ Qdrant（语义索引）
    │
    └─── Elasticsearch（全文索引）

用户提问
    │
    ▼
Query Rewriter（查询改写，可选）
    │
    ├─── Qdrant 语义检索 (top-20)
    │
    └─── ES 全文检索 (top-20)
    │
    ▼
Reciprocal Rank Fusion（结果融合）
    │
    ▼
bge-reranker-v2-m3（重排序，取 top-5）
    │
    ▼
LLM Gateway → DeepSeek-V3 / Qwen2.5（生成回答）
```

## 二、文档解析层

### 2.1 解析引擎选型

经过对比测试，选用 **pdfplumber** 作为主力 PDF 解析器，**PyMuPDF (fitz)** 作为 fallback：

| 解析器 | PDF 文本提取 | 表格提取 | 速度 | 兼容性 |
|--------|------------|---------|------|--------|
| pdfplumber | 优 | 优（原生表格检测） | 中 | 部分 CID 映射失败 |
| PyMuPDF | 优 | 中（需后处理） | 快（3x） | 兼容性好 |
| Tika | 良 | 良 | 慢（JVM 启动） | 格式支持最广 |
| unstructured | 优 | 良 | 慢 | API 友好但重 |

双引擎策略：pdfplumber 优先提取，失败段落（乱码率 >10%）回退到 PyMuPDF。实测乱码率从 30% 降至 5%。（详见 Bug 报告：部分 PDF 解析后中文乱码）

### 2.3 OCR 策略

对于扫描件 PDF，增加 OCR 预处理步骤：

- **检测逻辑**：如果 pdfplumber 提取的文本密度 < 10 字符/页，判定为扫描件
- **OCR 引擎**：PaddleOCR v4（中文优先）或 Tesseract 5.x（多语言场景）
- **性能影响**：OCR 处理速度约 2-3 秒/页，100 页文档需要 3-5 分钟（是纯文本 PDF 的 10-20 倍）
- **配置方式**：作为可选配置，由用户在上传时选择是否启用（默认关闭）

陈静建议在前端展示 OCR 置信度分数，让用户自行判断是否需要人工校对。

## 三、Chunk 策略

### 3.1 实验设计

基于启动会决议，我们在内部测试集（1200 篇文档，覆盖合同、技术手册、研究报告三类）上进行了 chunk 策略对比实验：

| 策略 | 描述 | F1 | MRR@5 | 平均 chunk 长度 |
|------|------|-----|-------|----------------|
| 固定 512 | 按 token 数固定切分 | 0.74 | 0.68 | 512 |
| 递归切分 | LangChain RecursiveCharacterTextSplitter | 0.78 | 0.73 | 480 |
| 段落分块 | 按自然段落边界切分 | 0.80 | 0.76 | 620 |
| **混合分块** | 段落分块 + 表格保护 + 重叠 | **0.82** | **0.79** | 550 |
| 语义分块 | 基于 embedding 相似度动态切分 | 0.81 | 0.78 | 490 |

### 3.2 最优策略：混合分块

最终采用的混合分块策略规则：

1. **段落分块**：以自然段落（\n\n 分隔）为基本单元，目标长度 512 tokens
2. **长段落切分**：超过 768 tokens 的段落在句号处切分
3. **短段落合并**：连续短段落（<128 tokens）合并直到达到目标长度
4. **表格保护**：表格、代码块作为独立 chunk，不切分
5. **重叠策略**：相邻 chunk 重叠 64 tokens，保持上下文连续性
6. **标题继承**：每个 chunk 前缀其所属的章节标题链（如 "第一章 > 1.1 概述 > "）

### 3.3 不同文档类型的效果

| 文档类型 | F1 | MRR@5 | 主要挑战 |
|----------|-----|-------|---------|
| 合同 | 0.85 | 0.83 | 条款编号需保留上下文 |
| 技术手册 | 0.83 | 0.80 | 代码片段和配置示例 |
| 研究报告 | 0.78 | 0.74 | 公式和图表引用 |

研究报告效果偏低，主要因为公式和图表引用在纯文本 chunk 中丢失了语义。二期考虑引入多模态 embedding 解决。

## 四、检索与重排序

### 4.1 双通道检索

- **语义检索**：Qdrant 向量检索，top-20
- **全文检索**：Elasticsearch BM25，top-20
- **结果融合**：Reciprocal Rank Fusion (RRF)，k=60

RRF 公式：`score(d) = Σ 1/(k + rank_i(d))`，其中 `rank_i(d)` 是文档 d 在第 i 个检索通道中的排名。

双通道相比纯语义检索，在精确匹配查询（如"合同编号 ZH-2026-0042"）上 MRR 提升 23%。

### 4.2 Reranker 性能

bge-reranker-v2-m3 在测试集上的效果：

| 配置 | MRR@5 | 延迟 p99（ms） |
|------|-------|---------------|
| 无 Reranker（直接取 top-5） | 0.71 | - |
| Reranker，输入 top-10 | 0.79 | 45 |
| Reranker，输入 top-20 | 0.83 | 82 |
| Reranker，输入 top-50 | 0.84 | 195 |

选用 top-20 作为最终配置（MRR 0.83 vs 0.84 仅差 0.01，延迟节省 58%）。

**已知问题**：Reranker 在表格密集型文档上效果显著下降（MRR 从 0.83 降至 0.65），原因是表格行被打散后语义不完整。解决方案见周报 W8 讨论。

## 五、LLM 生成层

### 5.1 Prompt 模板

```
你是一个企业文档智能助手。请基于以下参考内容回答用户的问题。

## 参考内容
{context}

## 用户问题
{question}

## 要求
1. 只基于参考内容回答，不要编造信息
2. 如果参考内容不足以回答，请明确说明
3. 回答中引用原文时标注来源文档和页码
4. 使用简洁、专业的语言
```

### 5.2 Token 配额机制

根据 CTO 评审意见（见评审意见_0207.md），需要实现按租户的 token 使用限额：

- 每个租户每月 token 配额（输入 + 输出）
- 实时用量统计和告警（80%/90%/100% 阈值）
- 超额后降级策略：切换到更小的模型或返回纯检索结果
- 管理后台可调整配额

## 六、待解决问题

1. 扫描件 PDF 的 OCR 质量不稳定，需要更多真实数据测试（关联：Bug报告_解析乱码.md）
2. 超长文档（>100 页）的 chunk 数量过多（可达 500+），检索效率下降——赵磊提议增加一层摘要索引（先对每个章节生成摘要，检索时先匹配摘要再深入 chunk）
3. 表格检索通道的设计方案待定（关联：周报 W8 reranker 表格问题）
4. 多轮对话的上下文管理策略（是否缓存历史 chunk 引用）
