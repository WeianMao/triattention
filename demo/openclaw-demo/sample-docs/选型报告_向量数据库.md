# 向量数据库选型报告

**作者**: 赵磊 | **日期**: 2026-01-29 | **版本**: v1.2

---

## 一、测试背景

根据 1/15 项目启动会决议，本报告对 Milvus 2.4、Qdrant 1.8、pgvector 0.6 进行全面 benchmark 测试。测试重点关注 DocMind 项目实际使用场景：企业文档的语义检索，数据规模预估 100 万-500 万条向量。

## 二、测试环境

| 项目 | 配置 |
|------|------|
| 服务器 | 8C32G，NVMe SSD 1TB |
| 操作系统 | Ubuntu 22.04 LTS |
| 数据集 | 100 万条 1024 维向量（bge-large-zh-v1.5 生成） |
| 数据来源 | 内部测试文档集（合同、技术手册、研究报告，共 12000 篇） |
| 索引类型 | Milvus: IVF_FLAT, Qdrant: HNSW, pgvector: ivfflat |
| 查询方式 | top-10 近邻查询，batch size = 1 和 batch size = 100 |

## 三、测试结果

### 3.1 核心指标对比

| 指标 | Milvus 2.4 | Qdrant 1.8 | pgvector 0.6 |
|------|-----------|-----------|-------------|
| 插入速度（万条/s） | 3.2 | 4.1 | 1.8 |
| 单条查询延迟 p50（ms） | 5 | 3 | 18 |
| 单条查询延迟 p99（ms） | 12 | 8 | 45 |
| 批量查询延迟 p99（ms） | 85 | 52 | 320 |
| 内存占用（GB） | 6.8 | 4.2 | 2.1（共享 PG 内存） |
| 磁盘占用（GB） | 8.5 | 5.8 | 12.3 |
| 召回率 @10 | 0.95 | 0.96 | 0.93 |
| 并发 QPS（10 并发） | 1200 | 1850 | 180 |

### 3.2 功能对比

| 功能 | Milvus 2.4 | Qdrant 1.8 | pgvector 0.6 |
|------|-----------|-----------|-------------|
| 混合检索（向量+标量） | 原生支持 | 原生支持（payload filter） | 需手动 JOIN |
| 分布式集群 | 原生支持（Pulsar/Kafka） | 支持（Raft 共识） | 需 Citus 扩展 |
| 多租户隔离 | Partition Key | Collection + API Key | Schema 隔离 |
| 增量索引 | 支持 | 支持 | 需 REINDEX |
| 备份恢复 | bulkinsert/backup API | Snapshot API | pg_dump |
| SDK 生态 | Python/Java/Go/Node | Python/Rust/Go/TypeScript | 任何 PG 客户端 |

### 3.3 运维复杂度

- **Milvus**: 依赖 etcd（元数据）+ MinIO（对象存储）+ Pulsar/Kafka（日志），最少 5 个容器。升级需要关注版本兼容性。社区活跃，文档完善。
- **Qdrant**: 单二进制部署，无外部依赖。集群模式需 3 节点起步。Docker 镜像仅 80MB，资源友好。文档清晰但集群部署最佳实践较少。
- **pgvector**: 作为 PostgreSQL 扩展，复用现有 PG 运维体系。但大规模场景需要精心调优（maintenance_work_mem、ivfflat lists 参数等）。

## 四、500 万条规模外推测试

在 100 万条基础上，使用合成数据扩展至 500 万条：

| 指标 | Milvus 2.4 | Qdrant 1.8 | pgvector 0.6 |
|------|-----------|-----------|-------------|
| 内存占用（GB） | 28.5 | 18.7 | 9.2 |
| 单条查询延迟 p99（ms） | 18 | 12 | 185 |
| 索引构建时间（min） | 45 | 32 | 210 |

pgvector 在 500 万规模下性能下降严重，单条查询 p99 升至 185ms，不满足 <50ms 的 SLA 要求。

## 五、结论与建议

**推荐方案：Qdrant 1.8**

1. **性能最优**：查询延迟、并发 QPS、插入速度均领先
2. **资源高效**：Rust 实现，内存占用仅为 Milvus 的 62%
3. **运维简单**：单二进制部署，无外部依赖
4. **召回率高**：HNSW 索引在测试集上达到 0.96 召回率

Milvus 功能更全面（特别是混合检索和分布式），但部署复杂度高，对一期项目而言过重。pgvector 适合 <50 万条的小规模场景，可作为轻量级客户的备选方案（见评审意见 #2）。

## 六、遗留问题

1. Qdrant 集群模式（3 节点 Raft）的生产部署方案需要王浩评估，特别是跨机房容灾
2. Qdrant 与 Elasticsearch 的查询结果合并策略（全文检索 + 语义检索双通道融合）需赵磊设计
3. pgvector 作为备选方案的切换成本评估（接口层需要做适配层抽象）
