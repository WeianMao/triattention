## 一、本周进展

### 后端（王浩）

1. **doc-parser 服务基本完成**：
   - PDF 解析准确率 92%（1200 篇测试集），Word 解析准确率 95%
   - 双引擎策略（pdfplumber + PyMuPDF fallback）已上线 dev 分支
   - 异步解析管线基于 Celery + Redis 实现，支持任务重试（最多 3 次）
   - 平均解析速度：纯文本 PDF 1.2 秒/页，扫描件 3.5 秒/页

2. **租户隔离方案已确认**：
   - 2/14 提交了隔离设计文档，CTO 已邮件确认通过
   - 方案：Qdrant payload filter（tenant_id 字段）+ 查询层强制注入租户条件
   - 文档存储使用 AES-256-GCM 加密，密钥由 HashiCorp Vault 管理
   - LLM 调用日志脱敏后存入独立的审计数据库

3. **CI/CD 管线搭建完成**：
   - GitHub Actions：lint → test → build → deploy（dev 环境自动部署）
   - Docker Compose 本地开发环境已验证（5 个核心服务 + Redis + PG + ES + Qdrant）
   - 生产 K8s Helm Chart 初稿完成，待 M3 阶段验证

### 算法（赵磊）

1. **RAG 管线端到端跑通**：
   - 全链路：上传 → 解析 → 分块 → 索引 → 检索 → 重排序 → 生成
   - 在 50 篇测试文档上 MRR@5 = 0.81（与离线实验 0.83 接近，符合预期）

2. **摘要索引原型完成**：
   - 使用 Qwen2.5-7B 对每个章节生成 100-200 字摘要
   - 两级检索：先查摘要索引（top-5 章节），再在命中章节内查详细 chunk
   - 初步结果：100 页文档查询延迟从 320ms 降至 180ms（44%↓），MRR 持平

3. **发现 Reranker 表格问题**：
   - 在表格密集型文档（财务报表、数据统计）上，Reranker MRR 降至 0.65
   - 根因分析：表格行被拆散后，单行"设计费 | 50万 | 含税"缺乏上下文
   - 临时措施：表格 chunk 保持完整，不参与 reranker 重排，直接进入 top-5
   - 长期方案讨论中：(a) 为表格设计独立的检索通道 (b) 用 LLM 对表格生成自然语言描述后再索引
