# 技术方案评审意见

**日期**: 2026-02-07 | **评审人**: CTO 周航 | **评审方式**: 现场评审（2小时）

## 一、总体评价

方案整体可行，架构清晰，团队对技术选型做了充分的实验验证（特别是 chunk 策略和向量数据库 benchmark）。以下几点需要修改后才能进入开发阶段。

**评审结论：有条件通过**

## 二、必须修改（Blocker）

### B1: 安全与数据隔离

当前方案的租户隔离设计不够清晰。具体问题：

1. Qdrant 的 Collection 级别隔离粒度太粗——同一 Collection 内不同租户的数据如何隔离？建议使用 payload filter + 租户 ID 字段，但需要验证性能影响。
2. 文档存储（原始文件 + 解析结果）必须加密，至少支持 AES-256-GCM 静态加密。
3. LLM Gateway 调用日志可能包含敏感内容，需要脱敏处理或独立存储。

**要求**：王浩在 2/14 前补充租户隔离设计文档，包括数据流图和安全边界说明。

### B2: 成本控制

DeepSeek-V3 虽然成本较低（¥1/百万 token），但不限量调用仍会导致成本失控。按照预估 DAU 500、每人每天 10 次问答、每次平均 8000 token（输入+输出）计算：

- 日均 token 消耗：500 × 10 × 8000 = 4000 万 token
- 月成本：4000 万 × 30 × ¥1/百万 = ¥1200/月（可接受）

但如果个别租户滥用（爬虫、批量调用），成本可能暴涨。

**要求**：增加 token 用量统计和按租户配额机制（见技术方案 5.2 节）。

### B3: 可观测性

当前方案完全缺少监控和告警设计。对于一个 AI 驱动的产品，可观测性是上线的硬性要求。

**要求**：接入 Prometheus + Grafana，至少覆盖以下指标：
- 文档解析延迟 p50/p99
- 向量检索延迟 p50/p99
- Reranker 延迟 p50/p99
- LLM 调用延迟 p50/p99 和 token 使用量
- 检索召回率（可通过用户反馈间接衡量）
- 系统整体 QPS 和错误率

## 三、建议优化（Non-blocker）

### S1: 摘要索引

chunk 摘要索引的想法很好（技术方案 6.2 节），建议一期就做，不要推到二期。理由：

1. 超长文档是企业客户的核心场景（100+ 页的合同和手册很常见）
2. 500+ chunks 的检索效率问题会直接影响用户体验
3. 摘要索引的实现复杂度不高，用 LLM 对每章节生成一段摘要即可

### S2: pgvector 备选方案

Qdrant 选型我同意（benchmark 数据很扎实），但建议同时保留 pgvector 作为轻量级备选。理由：

1. 小客户场景（<10 万条文档）pgvector 完全够用
2. 减少一个独立组件的运维成本
3. 为未来 PostgreSQL 生态的向量能力提升（pgvector 0.7 路线图很激进）留下升级路径

实现方式：检索层做一层适配接口，底层可切换 Qdrant/pgvector。成本不高。

### S3: 文档预览方案

建议用 **OnlyOffice** 而非 LibreOffice 转换。理由：

1. OnlyOffice 提供原生 Web 编辑体验，支持实时批注
2. LibreOffice 转 PDF 方案在复杂排版（合并单元格、嵌入图表）下经常丢格式
3. OnlyOffice 有 Document Server 的开源版本（AGPL），也有商业授权

**注意**：商业授权价格需要陈静尽快确认（后续周报 W8 显示报价 ¥15 万/年，超出预算）。

### S4: 错误恢复机制

文档解析是异步管线，需要考虑失败重试机制：

1. 解析失败的文档应标记状态，支持手动/自动重试
2. 部分解析成功（如 PDF 前 50 页成功、后 50 页 OCR 超时）应保存已成功部分
3. 建议引入简单的任务队列（Redis + Celery 或直接用 PostgreSQL advisory lock）

## 四、关于团队协作

1. 建议前后端接口定义使用 OpenAPI 3.0 规范，前端可以基于 spec 自动生成 TypeScript 类型
2. 算法和后端的代码审查应交叉进行，避免知识孤岛
3. 建议每周五下午 4 点固定 demo 会，各模块展示进展
