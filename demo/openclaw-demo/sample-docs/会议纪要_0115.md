## 一、项目背景

公司决定启动"智慧文档平台"（项目代号：DocMind）项目，旨在为企业客户提供基于大语言模型的文档智能处理平台。核心能力包括：

1. **文档解析与结构化**：支持 PDF、Word、PPT、Excel 等格式的自动解析，提取标题、段落、表格、图片等结构化信息
2. **智能摘要与问答**：基于 RAG 架构实现文档级别的摘要生成和多轮问答
3. **跨文档关联分析**：在多份文档间建立知识图谱，发现潜在关联
4. **协作与审批流程**：集成企业现有 OA 系统，支持文档的在线批注、审批和版本管理

项目一期目标覆盖前两项核心能力，预计 2026 年 4 月 30 日完成交付。二期将覆盖知识图谱和 OA 集成，目前暂不排期。

## 二、技术选型讨论

### 2.1 后端架构

王浩提出采用微服务架构，主要服务拆分为：

- **doc-parser**: 文档解析服务（Python，基于 unstructured + pdfplumber），负责将各种格式的文档转为统一的结构化 JSON
- **doc-index**: 文档索引服务（Python，基于 Elasticsearch + 向量数据库），承担全文检索和语义检索双通道
- **llm-gateway**: 大模型网关服务（Python，统一封装 OpenAI/本地模型调用），支持模型路由、限流、fallback
- **api-gateway**: API 网关（Go，基于 Gin 框架），处理鉴权、限流、请求路由
- **user-service**: 用户与权限管理（Go），支持 RBAC 角色模型和租户隔离

王浩补充说，服务间通信使用 gRPC + protobuf，API 对外使用 RESTful + JSON。部署方案初步考虑 Docker Compose 开发环境 + K8s 生产环境。

### 2.2 向量数据库选型

赵磊对向量数据库选型提出三个候选方案：

| 方案 | 优势 | 劣势 |
|------|------|------|
| Milvus 2.4 | 社区活跃，功能完善，支持混合检索 | 部署复杂（依赖 etcd/MinIO），资源占用大 |
| Qdrant 1.8 | Rust 实现，性能好，HTTP/gRPC 双协议 | 生态相对较新，集群模式文档少 |
| pgvector 0.6 | 与现有 PostgreSQL 集成，运维成本低 | 大规模性能不确定，无原生分布式 |

赵磊建议从插入速度、查询延迟、内存占用、召回率四个维度做 benchmark。张明同意，要求 1 月底前出具选型报告。

### 2.3 前端架构

李薇提出前端采用 Next.js 14 + TypeScript 方案，理由如下：

1. SSR 支持有利于 SEO 和首屏加载速度
2. App Router 提供更灵活的路由管理和 Layout 嵌套
3. Server Components 可以减少客户端 JS 体积，提升性能
4. 团队已有 3 个项目的 Next.js 经验积累，学习成本低

UI 组件库选用 Ant Design 5.x，图表库使用 ECharts。文档预览组件需要自研，支持 PDF 渲染（基于 pdf.js）和 Office 文档在线预览。Office 预览方案待定，候选有 LibreOffice 服务端转换和 OnlyOffice 在线编辑器。

陈静提出希望移动端也能使用。李薇建议一期先做响应式适配，二期再考虑原生 App 或 Flutter 方案。

### 2.4 模型选型

赵磊汇报了模型调研情况：

**摘要与问答模型**：
- 线上优先使用 DeepSeek-V3 API（成本低，中文长文档效果优秀，128K 上下文窗口）
- 备选 Qwen2.5-72B（私有化部署场景，客户要求数据不出域时使用）
- 本地开发/测试使用 Qwen2.5-7B（降低 GPU 成本，保持 API 兼容）

**Embedding 模型**：
- 中文场景首选 bge-large-zh-v1.5（维度 1024，MTEB 中文榜前五）
- 多语言场景使用 multilingual-e5-large（支持 100+ 语言，维度 1024）

**Reranker**：
- bge-reranker-v2-m3（跨语言重排序，支持中英文混合文档）

赵磊同时提出，RAG 管线中 chunk 策略对最终效果影响巨大，不同文档类型（论文、合同、技术手册）的最优切分方式差异很大。建议在技术方案中详细设计 chunk 策略，并准备对比实验数据。张明同意，要求赵磊在技术方案评审前完成实验。

## 四、风险识别

1. **模型效果风险**（高）：中文长文档的摘要质量难以保证，特别是专业领域（法律、金融）术语的准确性。需要充分的 prompt engineering 和人工评测。
2. **文档格式兼容性**（中）：企业客户的文档格式千差万别，扫描件 PDF 需要 OCR 预处理，手写体识别几乎不可能。
3. **性能风险**（中）：大文档（100+ 页）解析耗时较长，需考虑异步处理机制和进度实时展示。
4. **人力风险**（低）：前端团队目前只有李薇 + 1 名初级开发，后续若 UI 复杂度上升可能需要增援。
5. **合规风险**（低）：部分客户对数据存储有合规要求（等保三级），需提前了解并在架构设计中预留。
