# V2 方案调整计划（Fill-Hole + Thin Runtime Adapter）

- 更新时间：2026-02-23
- 状态：Active
- 适用范围：TriAttention_vLLM V2（vLLM 0.15.x / V1 Engine）

> 目的：把 2026-02-23 的方案共识固化为可接手执行的计划，避免后续继续在 patch-heavy 路径上叠复杂度。
> 本文是 `interface/V2_REFACTOR_EXECUTION_PLAN_2026-02-22.md` 的增量调整（补充并覆盖其中与执行顺序相关的部分条目）。

---

## 1. 本次方案调整的结论（TL;DR）

1. **目标模式只保留两种**：
   - `per_head`
   - `per_layer_per_head`
2. **`per_layer` 不是目标模式，也不是合理中间态**（禁止将其作为主线收敛步骤）。
3. **压缩执行主策略采用低搬运 fill-hole**（不追求保序，不做全量前缀重写）。
4. **允许使用 monkey patch，但必须是薄补丁（thin adapter）**，不能继续承担 decode 热路径主逻辑。
5. **复杂度应集中在压缩触发时**，压缩后的 decode 中间步只允许轻量适配，避免 CPU 拖累 GPU。

---

## 1.1 最终优先级（2026-02-23 负责人拍板）

后续方案取舍、代码实现、文档判断必须按以下优先级执行：

1. **HF 行为对齐（最高优先级）**
   - 必须优先对齐 HF 的 `per_head` / `per_layer_per_head` 行为与逻辑。
   - 为了性能或“代码看起来更干净”而改动 HF 语义，不允许。
2. **decode 推理效率（第二优先级）**
   - 压缩后的 decode 热路径必须尽可能轻量，优先避免 CPU 拖 GPU。
   - 复杂计算、推导、状态重算优先移到“压缩触发点”。
   - decode 路径代码改动与额外 metadata 引入都应最小化（能不引入则不引入）。
3. **工程实现取舍（第三优先级，平衡项）**
   - 在“不改 vLLM 源码目录”的前提下，平衡：
     - 非侵入式/规范式扩展（继承、可配置 class）
     - 开发复杂度与代码简洁性
   - 若使用少量 monkey patch / 函数替换能明显降低复杂度并更有利于前两项目标，则允许使用；
   - 若多种方案都能满足前两项目标，则优先选更规范、侵入更低的方案。

说明：

1. “允许 monkey patch”不等于“推荐 patch-heavy”。
2. 真正要避免的是：**decode 热路径 patch-heavy + step-local 全局状态 + 时序脆弱链路**。

---

## 1.2 本轮最终实现策略（拍板版，高层）

在上述优先级下，当前最终方案收敛为：

1. **HF 语义优先**
   - 主线只做 `per_head` / `per_layer_per_head`。
2. **布局层主线采用低搬运 fill-hole**
   - 不追求物理保序；
   - 只保证 keep 集合、K/V 对应关系、后续写入安全。
3. **运行时语义显式分叉**
   - 显式维护绝对位置 / 有效长度 / 写入位置（或等价表示）；
   - 不再依赖单一 `num_computed_tokens` 同时表达所有语义。
4. **触发在 scheduler 侧表达**
   - 支持后续“显存未满不压缩、显存压力到阈值才压缩”目标；
   - 固定间隔触发仅作为调试/回退策略。
5. **实现手段采用“少量核心替换 + 单点薄补丁”**
   - 不改 vLLM 源码目录；
   - 允许通过继承/替换类、函数替换、少量 monkey patch 达成；
   - 优先减少替换种类，避免实现风格杂乱。

补充约束（性能导向）：

1. decode 热路径上的 patch/adapter 不得承载复杂推导与大规模 metadata 组装；
2. 若某个 metadata 仅用于压缩触发步或调试，不应默认放入每步 decode 主路径；
3. 能用 request-local 持久状态增量表示的语义，不应在 decode 每步重新构造稀疏映射/字典。

---

## 1.3 对 R-KV/vLLM 的借鉴结论（纳入本方案）

R-KV/vLLM 的主要启发不是“某个压缩算法实现细节”，而是其开发方案对运行时语义的处理方式：

1. **把压缩后的关键语义显式放进 metadata / 状态流**
   - 例如：有效 KV 长度、占用 slot 映射、dropped token 计数等。
2. **让 scheduler / request 状态参与压缩语义闭环**
   - 触发与 request 状态不应只在 attention 内部隐式处理。
3. **避免依赖隐式全局状态的 step-local patch 链**
   - 这点与当前 V2 P0 问题高度相关。

不直接照搬的部分：

1. 不照搬其 fork 深改路径（我们仍坚持不改 vLLM 源码目录）。
2. 不把 attention forward 内压缩作为最终主线（会削弱 scheduler/allocator 可见性）。

## 2. 为什么要调整（面向接手同事）

当前 P0 问题的核心不是“算法公式错误”，而是：

1. 压缩后有效上下文状态与后续 attention 实际读取状态可能不一致（严重正确性风险）。
2. runtime override 链路过重（全局状态 + monkey patch + 时序假设），decode 热路径持续有 Python/CPU 介入（性能风险）。

这导致两个现象同时出现：

1. 准确率断崖式下降（例如内容重复、发散、循环输出）。
2. GPU 利用率下降，表现为 GPU 等 CPU。

本次方案调整的目标是：**在不放弃 HF 对齐与物理回收目标的前提下，主动降低实现复杂度与时序脆弱性。**

### 2.1 已合并的 P0 背景（替代独立 handoff 文档）

本次调整同时吸收了 2026-02-23 的严重退化问题排查结论（原独立 handoff 文档内容已并入本方案与 `OPEN_ISSUES.md`）：

1. 现象不是单纯“压缩效果变弱”，而是出现明显重复/发散/循环输出；
2. 高概率属于压缩后有效上下文状态不一致（读取映射/写入映射/长度语义错位）；
3. 最大嫌疑链路是 runtime input override 的全局状态 + monkey patch + 时序假设；
4. `enforce_eager=true` 更像放大镜，不一定是根因本身。

这直接推动了本文件第 1.1/1.2 节的优先级与实现策略拍板。

## 3. 新共识（必须遵守）

### 3.1 模式范围（强约束）

仅以下模式属于当前主线：

1. `per_head`
2. `per_layer_per_head`

明确非主线：

1. `per_layer`（不用于最终交付，也不作为过渡收敛路径）

说明：

1. 文档或任务拆分中若出现“先把 per_layer 跑稳再说”的表述，应视为过时方案，需改写。

### 3.2 压缩内存整理策略（主线）

主线采用：

1. **slot 级 fill-hole（低搬运）**
   - 将需要保留的 token 从 recent 区/尾部回填到前缀空洞；
   - 只搬运必要 token；
   - 不要求保留 token 的物理顺序与逻辑时间顺序一致。

明确不作为主线：

1. 全量前缀重写（例如“腾出全部 2048 再按顺序重写”）
2. 以“保序”为理由引入大规模无必要搬运

允许的优化：

1. page/block 级重排可作为优化 fast path（仅在表达能力足够时使用）
2. 无法纯 page 表达时，退回 slot 级 fill-hole / 必要 per-head 搬运

### 3.3 关于“保序”的结论（避免重复争论）

当前主线下：

1. **不把物理保序作为正确性要求**
2. 正确性要求是：
   - K/V 成对一致
   - attention 实际读到的 KV 集合与算法定义的 keep 集合一致
   - 压缩后后续写入不会覆盖本应保留的 KV

## 4. 对 vLLM 运行时语义的关键判断（本次调整的基础）

### 4.1 不能只靠“压缩时一次改一个状态”解决全部问题

原因（已按 vLLM 实际代码确认）：

1. vLLM decode 输入准备会在每步重新计算 `positions` 与 `seq_lens`
2. 且两者都由同一个 `num_computed_tokens` 派生（同源）
3. `slot_mappings` 又依赖 `positions`

因此在压缩后出现“绝对位置”和“有效上下文长度”分叉时：

1. 若完全不碰 decode 输入准备逻辑（哪怕轻量 patch），仅在压缩时修改单一状态变量通常不足以保持正确性

### 4.2 允许的简化方向（推荐）

可以采用：

1. **压缩触发时一次性更新 request-local 持久状态**（hack/重写状态是允许的）
2. **decode 每步只做薄适配**（轻量、稳定、低 CPU 开销）

这比当前方案更接近目标：

1. 不是“每步复杂推导 + step-local override”
2. 而是“压缩时重计算，解码时轻应用”

### 4.3 关于“压缩时一次 hack 能否直接解决”的最终判断

结论（用于后续避免重复讨论）：

1. **只在压缩点修改单一状态变量，通常不够**
   - 因为 `positions / seq_lens / slot_mappings` 在 vLLM 输入准备链路中并非独立语义。
2. **压缩点一次性更新多类持久状态 + decode 薄适配** 是可行方向
   - 这正是本方案采纳的路径。
3. 若完全不改 decode 输入准备逻辑（哪怕轻量 patch/adapter），高概率会重演语义错位 bug。

## 5. 新方案的总体形态（用于后续实现对齐）

### 5.1 总体原则

1. HF 语义层与内存布局层继续分离（沿用 V2 三层分离方向）
2. 复杂度放在压缩事件时，不放在 decode 常规步
3. decode 热路径插入逻辑必须尽可能轻量（O(batch) / 向量化优先）
4. decode 热路径新增 metadata 必须是“最小必要集合”，默认不扩张状态面

### 5.2 方案骨架（高层）

1. **语义层（Selector）**
   - 只负责 `per_head` / `per_layer_per_head` 的 keep 选择（HF 对齐）
2. **布局层（Compaction/Layout）**
   - 主路径：slot 级 fill-hole（低搬运）
   - 可选优化：可表达时使用 page/block 级重排
3. **运行时输入适配层（Thin Adapter）**
   - 负责把压缩后语义正确映射到 vLLM decode 输入准备链路
   - 可使用 monkey patch，但只做薄适配，不承载复杂推导

### 5.3 集成手段约束（实现风格约束）

允许：

1. 继承/替换 `scheduler`（优先，vLLM v1 有可配置入口）
2. 继承/替换 `worker/runner`（若需要）
3. 单点 monkey patch / 函数替换（用于弥补 vLLM 未暴露的替换入口）

禁止：

1. 直接修改 `vLLM` 源码目录文件
2. 同时维护多套风格混杂的 patch 链（难以维护）
3. 在 decode 热路径堆叠多个 step-local 全局状态 patch 机制

建议：

1. 替换方法尽量控制在 1-3 类，并由统一安装入口管理。

## 6. 必须显式维护的最小状态（目的导向，不规定实现细节）

以下状态是必须有的（名称可调整）：

1. **绝对进度语义**
   - 目的：保证 RoPE / position ids 按绝对时间前进
2. **有效缓存长度语义**
   - 目的：保证 attention `seq_lens` 反映压缩后有效上下文长度
3. **写入基址/偏移语义（slot base 或等价表示）**
   - 目的：保证新 token 写入压缩后应写的位置，不覆盖保留 KV

说明：

1. 这三类语义可以通过压缩点更新后的持久状态表达
2. 不要求沿用当前 `ACTIVE_*` step-local override 的实现方式
3. 不应为未来可能用到的场景预先扩增 decode 热路径状态

## 7. 对当前实现的调整方向（模块级目标，非代码细节）

### 7.1 要削弱/替换的路径（当前复杂度来源）

重点收敛对象：

1. `triattention_v2/effective_overrides.py`
   - 目标：不再作为 decode 每步主推导链
2. `triattention_v2/input_patch_state.py`
   - 目标：不再依赖模块级 step-local `ACTIVE_*` 作为长期事实源
3. `triattention_v2/runner_output_bridge.py` 中的 override 激活窗口
   - 目标：避免对执行时序的强假设

### 7.2 需要保留并收敛的能力

1. runtime input adapter 的“语义分叉”能力（必须保留）
2. fill-hole 低搬运 compaction 路径（作为主线策略）
3. reclaim / block table 同步能力（继续保留，避免回退为纯逻辑压缩）

## 8. 执行计划（详细版，供后续接手）

## A. 先做“可证伪”的一致性验证（优先级最高）

目标：

1. 不依赖 full-run，先验证压缩后状态是否一致

必须验证的命题：

1. 保留 KV 不会在后续若干步 decode 中被覆盖
2. attention 输入侧看到的有效上下文与 KeepPlan 一致
3. `seq_lens`、`positions`、`slot_mappings` 对应的是同一套压缩后状态

验收输出（至少一项）：

1. 最小状态一致性测试脚本/测试用例（可重复运行）

## B. 固化“压缩点更新持久状态”契约

目标：

1. 在压缩触发完成后，立即更新 request-local 状态，供后续 decode 轻量使用

要求：

1. 状态更新时机清晰（压缩成功后）
2. 状态与 reclaim/compaction 结果一致
3. 对失败/降级路径有明确定义（本步不压缩时如何处理）

## C. 将 decode 适配改为薄路径（性能主任务）

目标：

1. 把当前“每步复杂 override 计算”改成“每步轻量应用”

要求：

1. patch/adapter 仅负责应用已存在状态，不负责复杂推导
2. 避免大量 Python 字典遍历、CPU tensor 构造、时序敏感的全局激活窗口
3. 默认不引入额外每步 metadata 构造；若必须引入，应证明其必要性
4. 保留调试校验开关，但默认不进入热路径

## D. 以 `per_head` / `per_layer_per_head` 为主线完成布局与写入一致性

目标：

1. 不以 `per_layer` 作为阶段门槛

要求：

1. fill-hole 与后续写入行为在 `per_head` / `per_layer_per_head` 下可验证
2. 对无法 page 级表达的场景，允许 fallback 到必要的 slot/head 级搬运
3. 不为了“统一实现”回退到高搬运全量重写

## E. 再做性能回归（在正确性之后）

目标：

1. 验证 decode 热路径 CPU 开销下降、GPU 利用率恢复趋势

建议观测项：

1. 压缩后普通 decode 步的 CPU 开销（是否仍有重型 Python 逻辑）
2. 压缩触发步开销（允许较高，但必须可接受）
3. 端到端吞吐与稳定性（在等价配置下对比旧路径）

## 9. 明确的非目标（防止再次走偏）

本轮方案调整不追求：

1. 一次性移除全部 patch（允许薄 patch 作为现实折中）
2. 通过引入 `per_layer` 临时模式来“先跑通”
3. 为了保序重写全前缀 KV
4. 在问题未定位前直接用 full-run 反复试错替代一致性测试
