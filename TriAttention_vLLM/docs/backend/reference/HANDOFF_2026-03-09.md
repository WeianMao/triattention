## 一、本次修复的问题

### 1.1 压缩事件在 V1 async 路径丢失 (已修复)

**根因**: vLLM V1 的 `GPUModelRunner.execute_model()` 返回 `None`（async 模式，实际输出在 `sample_tokens()` 中产生）。原来的 `attach_execute_model_compression_events()` 遇到 `output is None` 直接返回，事件永远无法到达 scheduler 的 `update_from_output()`。

**修复**:
- `runner_output_bridge.py`: 新增 `scheduler_output` 参数，当 `output is None` 时将事件 `setattr` 到 `scheduler_output` 上
- `integration_monkeypatch.py`: `_patched_scheduler_update_from_output` 先从 `model_runner_output` 读事件，fallback 从 `scheduler_output` 读
- 可行原因: V1 单进程模式 (`UniProcExecutor`) 下 `scheduler_output` 是同一个 Python 对象，无序列化

### 1.2 batch_queue_dedup 阻止所有压缩 (已修复)

**根因**: `state.py` 的 `mark_trigger()` 设置了 `state.last_compression_step = step`。但 `mark_trigger` 在 `consume_runner_signals` 中被调用，这发生在 `execute_runner_compression_actions` **之前**。导致 dedup guard `signal.step - last_step <= 1` 永远为 True，所有压缩被跳过。

**修复**: `state.py` `mark_trigger()` 中移除 `last_compression_step` 更新。只有 `mark_compressed()` 和 `mark_compression_skipped()` 才更新此字段。

### 1.3 Worker 侧 block table reclaim 使用错误属性路径 (已修复)

**根因**: `worker_reclaim_sync.py` 使用 `base_runner.req_states`（不存在于 V1），导致 reclaim 静默跳过，`num_blocks_per_row` 永远不递减，最终 `append_row()` 时 shape 溢出崩溃。

**修复**:
- `worker_reclaim_sync.py`:
  - `req_id_to_index` 改为从 `base_runner.input_batch.req_id_to_index` 获取
  - `block_ids` 截断改为从 `base_runner.requests` dict 获取 req_state
- `runner.py` `_patch_scheduler_output_for_compressed_reqs`: 同样修复 `req_id_to_index` 路径

### 1.4 Scheduler 侧 block_reclaim=None 崩溃 (已修复)

**根因**: V1 batch-queue 模式下，scheduler 可能在 worker 已完成 block 截断后才收到事件，此时 `block_reclaim` 为 None。

**修复**: `scheduler.py` `_apply_compression_events` 中，当 `groups` 为空时 synthesize reclaim——直接用 `required_blocks` 截断 `req_to_blocks` 并调用 `free_blocks()`。

## 三、未解决问题: `vllm:kv_cache_usage_perc` 指标不下降

### 现象
生成过程中持续监控 `vllm:kv_cache_usage_perc`，使用率**只升不降**（0% → 24% 线性增长），即使 scheduler 侧 `free_blocks` 确认已调用。请求结束后瞬间归零。Demo UI "Used Tokens" 因此看不到压缩效果。

### 指标采样数据
```
t=2s:  0.57%   (正常增长)
t=20s: 7.37%
t=40s: 21.15%  (已经过多次压缩+free_blocks)
t=56s: 24.17%  (仍然线性增长，无下降)
t=58s: 0.0%    (请求结束，瞬间归零)
```

### 可能原因分析

**假设 A: 指标来自 worker 侧，不是 scheduler 侧 block pool**
`vllm:kv_cache_usage_perc` 可能从 GPU 侧的 block allocation 计算（如 `CacheEngine` 的 `gpu_cache` tensor 大小），而非 scheduler 的 `block_pool.free_blocks` 数量。scheduler `free_blocks` 只释放了 scheduler 的逻辑 block 追踪，GPU 物理内存并未释放。

**假设 B: 指标从 `num_gpu_blocks - num_free_blocks` 计算**
需要确认 `block_pool.free_blocks()` 是否真的把 block 加回了 `free_blocks` 计数器。可能 `BlockPool` 的 `free_blocks` 方法有 refcount 机制，block 只在 refcount=0 时才真正 free。

### 下一步排查方向

1. **确认指标计算来源**: 搜索 vLLM 代码 `kv_cache_usage_perc` 的赋值点，确认它读的是哪个 pool 的数据
   ```bash
   grep -rn "kv_cache_usage_perc" vllm/
   ```

2. **确认 free_blocks 效果**: 在 `_apply_compression_events` 的 `free_blocks` 调用前后，打印 `manager.block_pool.free_block_queue` 的长度变化

3. **如果指标来自 worker 侧**: 需要在 worker 侧也执行 block 释放——可能需要在 `worker_reclaim_sync.py` 中调用 worker 侧的 `CacheEngine.free_blocks()` 或类似接口

## 五、V1 架构关键路径参考

```
EngineCore.step_with_batch_queue():
  ┌─ scheduler.schedule()               # 创建 scheduler_output，附带 triattention_signals
  ├─ model_executor.execute_model(scheduler_output, non_block=True)
  │   └─ Worker.execute_model(scheduler_output)
  │       └─ TriAttentionModelRunner.execute_model(scheduler_output)
  │           ├─ consume_signals → execute_compression_actions
  │           ├─ apply_worker_block_reclaim_events    # worker侧 num_blocks_per_row 递减
  │           ├─ patch_scheduler_output_for_compressed_reqs
  │           ├─ GPUModelRunner.execute_model()       # 返回 None (async)
  │           └─ attach_events → setattr(scheduler_output, events)
  ├─ model_executor.sample_tokens()     # 产生实际 ModelRunnerOutput
  ├─ batch_queue.appendleft((future, scheduler_output, ...))
  │   ... (后续 step pop 时) ...
  └─ scheduler.update_from_output(scheduler_output, model_output)
      └─ _patched: 从 scheduler_output 读 events → _apply_compression_events
          └─ block_pool.free_blocks()   # scheduler侧 block 释放
```

**关键**: `scheduler_output` 在 UniProcExecutor 下是同一个 Python 对象（无序列化），但在 MultiProcExecutor 下不是。当前修复**仅适用于单进程模式**。
