## Bugs Found and Fixed

### 1. Off-by-One Error in TrigTableCache (CRITICAL)

**Location**: `triattention/kernels/triton_scoring.py`, line 106 in `TrigTableCache.get_trig_values()`

**Issue**:
- Index calculation `idx = (round_start // compress_interval) - 1` becomes negative when `round_start < compress_interval`
- No validation for `round_start` being a multiple of `compress_interval`
- Caused IndexError or returned incorrect cached trigonometric values

**Impact**:
- Runtime crash with IndexError when `round_start < compress_interval`
- Silent data corruption if negative index wraps around
- Incorrect scoring results leading to wrong token selection

**Fix**:
```python
# Before (BUGGY):
idx = (round_start // self.compress_interval) - 1
if idx < 0 or idx >= self.num_positions:
    raise ValueError(...)

# After (FIXED):
if round_start % self.compress_interval != 0:
    raise ValueError(f"round_start {round_start} must be a multiple of compress_interval")
if round_start < self.compress_interval or round_start > self.compress_interval * self.num_positions:
    raise ValueError(f"round_start {round_start} out of range")
idx = (round_start // self.compress_interval) - 1
```

### 2. Invalid Dtype for position_indices (MEDIUM)

**Location**: `triattention/config.py`, lines 157-162 in `TriAttentionConfig.__post_init__()`

**Issue**:
- `torch.bfloat16` listed as valid dtype for `position_indices_dtype`
- bfloat16 has only 8-bit mantissa, causing precision loss for positions > 256
- Position indices should be integers, not floating-point

**Impact**:
- Silent precision loss for long sequences (seq_len > 256)
- Incorrect position values used in scoring
- Potential misalignment between position indices and actual token positions

**Fix**:
```python
# Before (BUGGY):
valid_position_dtypes = {torch.int32, torch.bfloat16, torch.int64}

# After (FIXED):
valid_position_dtypes = {torch.int32, torch.int64}
```

**Test Coverage**:
- `TestConfigDtypeFix::test_valid_position_indices_dtypes`
- `TestConfigDtypeFix::test_invalid_position_indices_dtype_bfloat16`
- `TestConfigDtypeFix::test_invalid_position_indices_dtype_float32`

### 3. Division by Zero in normalize_scores (MEDIUM)

**Location**: `triattention/utils.py`, lines 193-206 in `normalize_scores()`

**Issue**:
- When all scores are constant (std=0), division by `std + eps` produces Inf/NaN
- `eps=1e-8` is too small to prevent division by zero when std is exactly 0
- No handling for edge case of constant scores

**Impact**:
- NaN/Inf propagation through TopK selection
- Random/undefined behavior in token selection
- Potential crash in downstream operations

**Fix**:
```python
# Before (BUGGY):
mean = scores.mean(dim=-1, keepdim=True)
std = scores.std(dim=-1, keepdim=True)
return (scores - mean) / (std + eps)

# After (FIXED):
mean = scores.mean(dim=-1, keepdim=True)
std = scores.std(dim=-1, keepdim=True)
std_safe = torch.where(std < eps, torch.ones_like(std), std)
return (scores - mean) / std_safe
```

### 4. Missing disable_mlr Implementation (MEDIUM)

**Location**: `triattention/kernels/triton_scoring.py`, line 274 and `speckv_scoring()` wrapper

**Issue**:
- `disable_mlr` parameter documented but not implemented in Triton kernel
- Warning issued but kernel still uses MLR formula
- Misleading behavior - users expect MLR to be disabled but it's not

**Impact**:
- Incorrect results when `disable_mlr=True` is used
- Confusion about actual scoring behavior
- Inconsistency between PyTorch and Triton implementations

**Fix**:
```python
# scoring.py - Automatic fallback to PyTorch
if config.use_triton_scoring and not config.disable_mlr:
    return compute_scores_triton(...)
else:
    return compute_scores_pytorch(...)

# triton_scoring.py - Clear error instead of silent warning
if disable_mlr:
    raise NotImplementedError(
        "disable_mlr=True is not supported in Triton kernel. "
        "Use compute_scores_pytorch() instead by setting config.use_triton_scoring=False."
    )
```

**Test Coverage**:
- Existing tests in `test/test_scoring_correctness.py` cover fallback behavior

### 5. Unused position_indices Parameter (LOW)

**Location**: Multiple locations in `scoring.py` and `triton_scoring.py`

**Issue**:
- `position_indices` parameter deprecated but still allocated as dummy tensor
- Wastes GPU memory (4 bytes per token * batch_size * seq_len)
- Confusing API - parameter documented as deprecated but still required

**Impact**:
- Minor memory waste (negligible for typical batch sizes)
- API confusion for users

**Status**:
- Documented in code comments
- Minimal dummy tensor allocated (1 element) to satisfy kernel signature
- Not fixed to maintain API compatibility
- Will be removed in future API version
